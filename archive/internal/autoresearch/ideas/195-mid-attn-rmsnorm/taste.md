# Taste log — 195 mid-attn-rmsnorm

## r1 — 2026-06-15 — verdict: revise
- **Step-0 byte-identity is muddled — commit to one formulation.** The pitch cycles through THREE init strategies and admits the obvious one (`γ_h=1`, plain RMSNorm) is *not* bit-identical to baseline. The two compensating forms (auto-init from first-batch stats; inverse-scale-then-rescale) are non-trivial and the first one requires a *frozen* baseline_rms_estimate the design sketch never pins down. Pick ONE and show the math in two lines: (a) which scalar gets applied, (b) what value it has at step 0. If you can't get bit-identity with a 1-line guarantee, say so explicitly and use a *non*-bit-identity ctrl-comparison protocol (two ctrls bracketing the post-init step-0 RMS) — that's a definition-gate decision but the taste gate needs the lever's identity story to be *crisp* or the A/B is noisier than the bar.
- **The per-head γ_h axis is exactly the closed null axis.** The closed.md evidence is unambiguous on per-head attention-shape levers at 0.94M/12L/4H: 155 per-head temp (Δ=−0.0063 inside band), 152 per-head bias (null), 160 post-AV gain (null Δ=−0.0023). 195's *only* learnable is γ_h — the per-query RMS component has zero parameters, it's a fixed normalization. So the optimizer sees a per-head scalar over a fixed-reshape of the scores, and three per-head scalars have already closed null. The pitch needs a *strong* argument for why RMSNorm-on-scores breaks the per-head-scalar-null pattern. Suggestion: frame the bet as a test of *score-distribution shape* (per-query variance of scores), not as a test of γ_h. If the value is in the fixed per-query normalization, say so and downweight the γ_h story.
- **Differentiate the bet from 190-per-layer-qk-norm (just accepted).** 190 is the *granularity* axis of 016's WIN (per-block vs per-head γ). 195 is a *placement* axis (pre-product vs post-product QK^T). Both claim attribution-insight value. If both go to definition, the queue will run two "what is 016's WIN actually about?" A/Bs back-to-back — that's not a portfolio, it's a duplicate. Make 195's *unique* contribution sharp in one sentence. Suggested framing: "195 is the ONLY lever in queue that tests *post-product* QK normalization; 016 tests *pre-product*; 190 tests *granularity*; together the three partition the QK-axis attribution." If you can't draw that triangle, the lever overlaps.
- **Compress the design sketch to mechanism-level only.** The current sketch (compute RMS, divide by RMS, multiply by γ_h, plumb layer_idx, freeze baseline_rms) is *implementation* — that's the code gate's job. For taste, name the lever in 3 lines: "per-query RMSNorm over key axis, per-head learnable γ_h, init TBD." Drop the first-batch stat collection detail; if the definition gate chooses a non-bit-identity init, that's their call.
- **Frame the bet as "we expect val loss to drop by X" not as "we want to close an attribution question."** The current "Why it's worth a slot" is *attribution-only* — a clean null result would close the placement axis but the win case is "we matched 016's Δ=−0.0138." Name the expected magnitude. If the bet is "Δ ≤ −0.01 at tiny1m3m" then say so; if it's "Δ between 0 and −0.01" then the lever is null-only and the cost (per-query compute + a third pre-softmax QK variant in queue) is harder to justify. The pipeline screen needs a positive WIN hypothesis, not just an information-value claim.
- **Transfer note.** Placement-of-normalization is scale-free in principle; transfer-risk: low is correct. The T×T per-query RMS compute is a real cost (~2048× overhead per head on the per-query axis) and the marginal value at 135M (where per-head scalars already don't bind per the closed pattern, *unless* the RMSNorm-reshape changes the math) is unproven. Acceptable to defer to 135M with a "tier-mismatch hedge" note in the bet.
