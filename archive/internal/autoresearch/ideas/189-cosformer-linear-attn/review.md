## r1 — 2026-06-15 — verdict: revise

Mechanism is real, source is current (Qin et al. 2022, arXiv:2202.08791), niche fits (kernel replacement, not HP), scope correct (tiny1m3m/seed 42), pass-bar tightened appropriately, and the auxiliary entropy diagnostic is well-chosen. **Findings to address before plan/spec:**

- **Co-twin with 189-cos-attn is a coordination hazard, not a duplicate.** A sister idea `189-cos-attn` is already in `implementing` (status: implementing, 2026-06-15T08:31:50Z) — it cites the same Qin et al. 2022 paper but proposes a *different* lever: L2-normalized pre-softmax scores `(Q/||Q||)·(K/||K||)^T` with standard softmax kept. The two are **mechanistically distinct**: 189-cos-attn = bounded-pre-softmax scores, softmax stays; 189-cosformer-linear-attn = softmax removed, replaced with cosFormer feature map `φ(x) = exp(γx)·cos(x)`, linear-time KV matmul. Implementer must not conflate the two. **Add to plan.md §1 a one-paragraph "Distinct from 189-cos-attn" gate** that names both levers and asserts the implementer is building the linear-time kernel-replacement form, not the bounded-softmax form. Without this gate, the two near-twin plans risk being merged or one being skipped.

- **Closest empirical prior is 004-retnet, not 148-focal-mod.** The post-null info value in the plan names 148 (additive context) and 008 (gated linear) as siblings. 008 was a taste-reject (off-niche, never ran). 148 was additive context (softmax stays). The lever that shares the **structural template** with 189 is **004-retnet-retention**: O(T·d_k²) linear-time attention with a feature map, null Δ=+0.04 wrong-sign at 0.94M. 004's φ was `elu(x)+1`; cosFormer's φ is `exp(γx)·cos(x)`. The diffuse-kernel argument (plan's §"Sharp prediction") must be tied to **why a cosine feature map should beat elu+1 at 0.94M** — at this tier the linear-vs-quadratic complexity bet is invisible (T=2048 is too short), so the lever reduces to a *kernel-shape softmax replacement*. 004 already nulled on that exact bet. The plan should rewrite the post-null section to name 004 as the direct prior and explicitly argue against the null-from-004 generalization (e.g. "elu+1 is non-negative and concentrates mass on the max; cos+sin is bounded and oscillates, so it diffuses mass on a wider band — a stronger fit for low-data generalization at 92 update steps"). As written, the plan reads 148 (additive context) and 008 (off-niche) as the empirical priors, neither of which is the right one.

- **Bit-identity verification is on a toy, not the actual model.** The miner's "max|Δ output| < 1e-5 with seed 42, one-block d=64, T=512" is a synthetic check. The actual tiny1m3m Q, K are `nn.Linear` projections of the residual stream; per-row ||Q||_2 and ||K||_2 are O(1/sqrt(d_k)) under the std-0.02 qkvo_proj init (not 1, as the toy assumes). cos(Q) ≈ 1 − Q²/2 holds elementwise, but the second-order statistics of Q'K'^T differ from the toy. **Plan.md must specify a step-0 check at the actual model**: `trt = build(use_cosformer=True); ctrl = build(use_cosformer=False); fp32_max_abs_diff(trt(x), ctrl(x)) < 1e-6` for the same input, same seed, with the real MHA class, real W_Q/W_K init, real n_heads=4, real d_k=16, real n_layers=12. The mean-pool reduction is still correct (cos(Q)·cos(K)^T ≈ 1 at small Q,K), but the test must be against the real model. A 1e-5 toy bound on a 1-block synthetic is not the project standard (which is 1e-6 fp32 max-abs-diff at the real step-0).

- **Denominator Z = Q'·K'^T is mandatory — bind it in the plan.** The plan says "the cosFormer paper's normalization helps; without it the lever is essentially a global mean-pool". This is the entire point of cosFormer. **Bind the denominator in plan.md as non-conditional**: the spec must say "compute `KV = K'^T · V` (B,H,d_k,d_k), then `out = Q' · KV`, then `out = out / (Q' · K'^T).clamp_min(1e-6)`" with no flag to skip the denominator. If the implementer can choose to drop the denominator (as the plan's wording implies), the lever silently nulls to mean-pool. Make it a hard branch — same shape as the `use_qk_norm` flag, which forces the manual path because the mechanism requires it.

- **Causal mask must be specified.** Linear attention needs explicit causal handling: for query at position t, only keys at positions s ≤ t should contribute. The existing `use_linear_attn` path at `models/layers.py:4301-4332` handles this with `cumsum` indexed by `[end_idx, start_idx]` (a prefix sum clipped to the window). cosFormer's Q'·(K'^T·V) formulation leaks future tokens by default (the K'V matmul uses ALL keys). The plan must specify the causal form: either (a) per-query `out_t = sum_{s≤t} φ(Q_t)·φ(K_s)·V_s / sum_{s≤t} φ(Q_t)·φ(K_s)` done position-by-position (O(T²·d_k), defeats the linear-time claim), or (b) compute `prefix_kv` with a causal mask on the K'V matmul so each prefix position only includes s ≤ t, or (c) the "causal linear attention" trick: `out_t = Q_t · (cumsum(K'^T·V))[t] / (Q_t · (cumsum(K'))[t])` — the prefix-sum form that gives true linear-time *and* causal. The plan should pick (c) and name the line where the cumsum lives (likely a new branch in `MultiHeadAttention.forward` analogous to lines 4314-4330). Without this, the spec is incomplete and the implementer is likely to leak future tokens (180-qk-logit-conv pattern).

- **Forward-path placement: extend `use_linear_attn` or add a new flag.** The plan says "add a `cosFormerAttention` module alongside the existing softmax attention". The existing convention is to add a flag to `MultiHeadAttention` and a new branch in `forward` (see `use_qk_norm`, `use_fire_pe`, `use_cope`, `use_fox`, `use_softpick`, `use_entmax`, `use_ssmax`, `use_linear_attn` — all flags on the MHA class, no new module). **Pick the convention**: add `use_cosformer: bool = False` to `MultiHeadAttention.__init__` and a new `elif self.use_cosformer:` branch in `forward` (placement: right after the existing `elif self.use_linear_attn:` block at line 4301). Pass `cosformer_gamma_init: float = 0.0` as a separate kwarg. The new `cosFormerAttention` *module* would be a regression from the project pattern.

- **The "γ=0 is bit-silent" claim should be tightened to "γ=0 ⇒ mean-pool over causal prefix".** The math is right but the verbal claim in the plan says "mean-pool(V)" — at causal position t, both softmax-with-small-logits and cosFormer-with-γ=0 output the *cumulative mean* of V over the prefix s ≤ t, not the global mean over the full T. A bit-identity check at position t=0 with mask-only-at-0 is closer to a uniform scalar (all-zero V averaged is 0) — implementer should verify against the *cumulative* mean, not the global mean. The numerical verification should be `assert max_abs_diff(trt_out[t], cummean(ctrl_out)[t]) < 1e-6` for t in 0..T-1, where `cummean(ctrl_out)[t] = mean(ctrl_out[:t+1])` over the causal prefix. (For t=0, that's just ctrl_out[0].)

- **Param count: 12 γ scalars, init 0, in `R^{n_layers}`.** Plan is correct: one scalar per block, init 0, learned. Should be a single `nn.Parameter(torch.zeros(n_layers))` on the model (following the `layer_temperature` pattern in line 795 — "the parameter lives on the model, not the MHA, so the optimizer sees ONE nn.Parameter"). Plan should say this — currently it just says "1 γ scalar per block × 12 blocks = 12 params, negligible" without specifying the layout. Without the explicit "parameter lives on MinimalLLM" gate, an implementer might create 12 separate Parameters on the MHA, which means 12 entries in the optimizer's param groups (not the project's flat layout).

- **"Sharp prediction" mechanism should cite the kernel-shape literature, not just the diffuse-kernel intuition.** The current prediction (cosine feature map is bounded, softmax's exp is sharp → cosine diffuses mass more evenly → better generalization with limited tokens) is plausible but asserts a quality claim the cosFormer paper *itself* does not make (Qin et al. 2022 reports parity with softmax, not a win). The diffuse-kernel argument is the miner's own, not from the source. That's OK — the protocol allows it — but the plan should mark it as the *mechanistic bet* and add a reference to the closest literature analog (e.g. Choromanski et al. 2021 Performers FAVOR+ for the "feature map is kernel-approximable" framing, or Yang et al. 2024 gated linear attention for the "more diffuse than softmax generalizes better at low data" framing). One sentence is enough.

- **Auxiliary diagnostic thresholds are tight but defensible.** "Attention entropy at step 10 ≥ softmax's entropy × 1.10" — this is the correct axis (catches degenerate-φ init collapse) and the threshold is reasonable. Keep.

- **Niche fit, transfer-risk: med, source citation, scale evidence — all clean.** Source is real (Qin et al. 2022, NeurIPS, arXiv:2202.08791, validated at 100M-1.5B on language modeling). Transfer-risk: med matches the validation scale. Scale evidence section names the largest scale (GPT-2-small ~125M). No fabrication.

**Summary**: 8 actionable findings, all addressable by the reviser without re-running taste. None are reject-level. The mechanism is sound, the source is current, the niche fits, the pass-bar is tightened appropriately, and the auxiliary diagnostic catches the relevant failure mode. Revise.
