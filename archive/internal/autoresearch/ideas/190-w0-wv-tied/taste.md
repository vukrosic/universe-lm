# 190 — Tied W_O / W_V (taste log)

## r1 — 2026-06-15 — verdict: accept
- **Sharp bet present.** One sentence: "tied W_O/W_V is a 0.43% param reduction that enforces a norm-preserving (W_V·W_Vᵀ) round-trip; a null tells us even soft tying is too constraining at this tier, a win gives a meaningful efficiency-accuracy trade-off." That's the crisp-bet bar.
- **Mathematically distinct from the closed tied-QK axis.** Tied QK closed because it forced Q and K to share a *role* (query direction vs key direction must differ). W_O ↔ W_V tying is a *symmetry* constraint on the value round-trip — different projection shapes, different mechanism in the attention forward, no role collision. The miner's defense against the closed-twin is well-argued, not handwaving.
- **Null is still informative.** If it fails at 0.94M/3M-tokens, the lesson is "soft tying on the attention path doesn't pay off at this scale either" — a *cleaner* null than tied-QK's collapse, because it would isolate the *softness* axis. Both win and null move the picture.
- **Portfolio not crowded.** Active queue is heavy on attention-shape variants (cos-attn, relu-attn, topk-attn, qk-clamp, rope-phase-offset, mlp-token-mixer). The only weight-sharing lever in flight is the parallel-AI tied-qk work in the working tree. Plenty of room; this is not a 5th-of-the-family pitch.
- **Niche fit is clean.** Mechanism not HP, identity-init-able (shared `_init_weights` → same seed → same byte values at step 0, since they point at the same Parameter), tiny1m3m-runnable. Transpose-tied variant (`W_O = W_Vᵀ`) is shape-portable, so the lever won't die when d_k·H ≠ d_model at 135M (the miner correctly notes the direct-tied form is square-attention only).
- **Scale evidence is real and on-point.** ALBERT validates cross-layer sharing at 12M-235M; T5 tied-embeddings validate the tied-projection *philosophy* at 60M-11B. Transfer-risk: low is correctly tagged; the lever is a regularization axis, not a scale-specific trick.
- **Leverage is honest.** 0.43% param savings is small, but the *quality* effect (symmetric round-trip = bounded attention output magnitude) is the actual bet. Param savings are a bonus, not the point — and the miner doesn't oversell them.

### Notes for the definition gate
- **Pick one variant for the run, don't A/B both.** Transpose-tied (`W_O = W_Vᵀ`) is the canonical choice: shape-portable to 135M, doesn't require d_k·H = d_model, and the miner's "symmetric round-trip" intuition is the cleanest. The direct-tied variant is square-attention only — keep it as a follow-up if transpose-tied wins, not as a parallel A/B.
- **Step-0 byte-identity needs an explicit initialization story.** With the same `_init_weights` applied once and the Parameter shared, the tied model is bit-identical to itself at step 0 (trivially). But the spec wants "step-0 ≈ baseline" — meaning the *un-tied* model and the *tied* model should be indistinguishable at step 0. Since the un-tied model has two independent random matrices and the tied model has one used twice, they're not bit-identical; they're "in distribution" equivalent. The miner's note that this is the same category as 175-alibi (slope-init 0 produces a non-zero forward) is the right framing. Definition gate should write the *un-tied* model first, snapshot its (W_V, W_O) pair, then overwrite *both* slots with the same Parameter to construct the tied model — gives a clean "step-0 un-tied forward vs step-0 tied forward" delta for sanity.
- **Cache reference is the right one.** Champion 6.24 / cache baseline 6.40; WIN bar −0.005, two-ctrl rule in force.
- **No conflict with parallel-AI tied_qk work.** Different flag (`use_tied_wo_wv` vs `use_tied_qk`), different projection. Definition gate can proceed without coordination beyond the standard "git diff before editing" check.
