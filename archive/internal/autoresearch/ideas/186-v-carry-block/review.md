# 186 — Within-Block V-Carry (review)

## r1 — 2026-06-15 — verdict: approve

- **Source citations check out.** Katharopoulos et al. 2020 (arXiv:2006.16236) is the canonical linear-attention / "Transformers are RNNs" paper — recurrence `S_t = S_{t-1} + k_t ⊗ v_t` is real, well-validated at 1B+ via Performer / Linear Transformer / RWKV. Shi et al. 2024 (arXiv:2407.06641, rebased attention) is the 154 cite. `transfer-risk: med` matches the source evidence.

- **Mechanism is a mechanism, not a hyperparameter.** Per-head learnable scalar `α_h = tanh(α_raw_h)` with `α_raw_h = 0` init ⇒ `α_h = tanh(0) = 0` exactly ⇒ `V_h_t + 0 · V_h_{t-1} = V_h_t` ⇒ forward and backward graphs byte-identical to baseline at step 0. The tanh parameterization is the load-bearing trick that keeps `|α_h| < 1` strictly and prevents geometric-sum explosion (`1 + α + α² + …` is finite iff `|α| < 1`). +48 params total (H=4 × n_layers=12), ~+0.005% of 0.94M — negligible.

- **Distinct from every closed V-axis lever on the right axes.** 154 = static rebase (K and V, fixed). 168 = cross-block AV carry (post-W_O, fixed α=0.999). 164 = cross-block Q carry (Q side, fixed α). 021 = cross-block V residual (V side, identity init). 186 = within-block V carry (V side, per-head learned α_h). All four axes of variation (placement, tensor, parameterization, init) differ. The 168 null is a real prior but a different lever form — 168 had fixed α (no per-head axis) and was cross-block (needs 12 sequential blocks to develop signal).

- **Implementable in <200 LoC against the real files.** Recipe matches the existing 164/168 carry pattern (per-block scalar carry + .detach() stash). The V-reshape site (`models/layers.py:2899` — `V = V.reshape(batch_size, seq_len, V_n_kv_heads, self.d_k)`) is the natural insertion point for the per-head α carry, right before the `[B, H, T, d_k]` transpose. The per-head α_h allocation is `self.v_carry_alphas = nn.Parameter(torch.zeros(n_heads))` (mirrors 155-per-head-temp's per-head shape). Recipe's stated ~50 LoC is realistic.

- **Implementer-actionable (not blockers, just notes for the reviser/code gate):**

Use a vectorized associative scan / `torch.cumsum`-style trick (`cumV_t = Σ_{s≤t} α^{t-s} · V_s`) or the fla-org linear-attention scan kernel. Document the choice in the plan; tag the implementation with a comment.
  - **Step-0 byte-identity across all 12 blocks.** The α_h parameter is per-head but the MHA flag is per-block. Verify `max_abs_diff(trt_step0_logits, ctrl_step0_logits) == 0.0` for `T=2048` and ALL 12 blocks — a single-block α init typo will break bit-identity for that block's output even if the others pass.
  - **Per-head α_h readout at end-of-training is the interpretability signal.** Without it, a null is uninterpretable (did the optimizer not move α? did it move to 0? did it diverge per-head?). Add `trt.attention.v_carry_alphas.detach().cpu().abs()` to the run artifact; this is the secondary signal that turns a clean null into a mechanistic close (vs a 168-style "didn't fire" black box).
  - **Padding / causal boundary.** At `t=0`, the carry has no `V_{t-1}` — the recipe's `V_new[:, :, 0, :] = V[:, :, 0, :]` (zero carry at the left edge) is correct, no special token. Verify this doesn't break step-0 byte-identity.
