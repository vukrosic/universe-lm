## r1 — 2026-06-15 — verdict: revise

- **Mask magnitude / lever continuity under-specified — blocking.** `mask = causal_mask * (1 − γ_h)` with `causal_mask = float("-inf")` (the existing repo convention per `models/layers.py:89,159,169,172` and the closed `_hybrid_head_mask`/`_multiscale_mask` paths at `models/layers.py:3442,3449`) makes the lever effectively all-or-nothing: for any γ_h < 1, `(1 − γ_h) · -∞ = -∞`, so the upper-triangle stays masked until γ_h hits 1 exactly (sigmoid can saturate but not reach exactly 1). The plan's "progressively bidirectional" framing is wrong under -∞. **Fix:** in the plan, use a finite `causal_mask_value` (e.g. `−1e4` or `min_dtype`) instead of `-∞`, so `(1 − γ_h) · causal_mask_value` is a real gradient and the per-head mix is continuous and interpretable. Alternative: explicitly re-frame the lever as a per-head *causal-vs-global switch* (no middle ground), and document that the optimizer gets a step-function signal. Either way, write the chosen interpretation into the plan and the falsifiable bar.
- **No concrete pass/fail bar at tiny1m3m — blocking.** `idea.md` never names a target val-loss Δ or a numeric pass-bar tied to the cache baseline (6.4394 ± 0.04, three-ctrl cluster per `autoresearch/baseline-cache.json` plus recent closed runs 152/155/162/166). Box noise is ±0.01, so a sub-0.01 effect is unresolvable at this tier and must be flagged inconclusive, not promoted. **Fix:** add a `## Pass/fail bar` section with (a) expected Δ-val range (the hypothesis predicts a small *or* null effect given the per-head-attention-shape trio that just closed null at 152/155/166 — call it e.g. `Δ ∈ [−0.01, +0.005]`), (b) a numeric pass-bar (e.g. `trt_val ≤ ctrl − 0.01` for WIN, `|trt_val − ctrl| ≤ 0.01` for NULL, `≥ +0.01` to flag regression), and (c) explicit treatment of the train/inference γ flip in that bar (the next finding).
- **Inference-time γ_h schedule not specified — blocking.** `taste.md` explicitly punted this to the definition gate: "either parameterize the eval-time override explicitly or document the inference-time γ_h = 0 schedule as part of the run." `idea.md` does not. At inference the model has no future context; trained γ_h > 0 is a train/inference distribution shift the way nothing else in the closed batch is. **Fix:** pick one and write it into a `## Inference schedule` section. Recommended: keep γ_h as trained for both train and eval (so the WIN/NULL verdict measures the actual deployment behavior, not a contrived causal override). If you instead force γ_h = 0 at inference, you must also pre-register a paired sanity run with γ_h frozen at 0 throughout training, to disambiguate "train-time leakage helps" from "the trained head can re-route around the mask."
