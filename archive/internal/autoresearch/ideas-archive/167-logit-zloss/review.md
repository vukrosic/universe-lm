- **Source is real, current, well-attributed.** Chowdhery et al. "PaLM: Scaling Language Modeling with Pathways" (arXiv:2204.02311, JMLR 2023) introduced z-loss as `λ·log(Z)²` (Z = partition function). Used in PaLM 540B with `λ=1e-4`, then adopted by LLaMA 2, OLMo 2, Gemma. "StableLM" / "Gemma" stability reports are accurate context for the secondary claim. Not a fabrication. arXiv id resolves, authors plausible, mechanism correctly described.
- **Mechanism is a mechanism, not an HP lever.** The lever adds an auxiliary *loss term* that penalises the partition function — structurally distinct from any LR/schedule/init knob. At `zloss_weight=0.0` the auxiliary term is exactly 0 and the loss is bit-identical to baseline CE (the verify path is `torch.logsumexp(logits, dim=-1).pow(2).mean()` — confirmed correct, NOT the misread `sum_v logits_v` the spec warns about). Step-0 identity is verifiable and load-bearing.
- **Already 80% wired in the trainer.** Surprise: `training/trainer.py:1460-1465` and `1548-1552` (AMP) + `1654-1658` (CPU) already read `getattr(config, "use_z_loss", False)` and `getattr(config, "z_loss_lambda", 0.0)` and add `z_loss_lambda * (logits.logsumexp(dim=-1) ** 2).mean()` to the total loss (line 1599/1702). The defaults collapse to `z_loss = logits.new_zeros(())` and the baseline path is bit-identical. **The implementer's job is therefore small: add `use_z_loss: bool = False` and `z_loss_lambda: float = 0.0` to `configs/llm_config.py` (or a YAML / CLI override path) and wire a tiny1m3m config that sets `z_loss_lambda=1e-4`.** No `models/layers.py` change needed, no Muon/optimizer routing change, no AMP-matmul fork.
