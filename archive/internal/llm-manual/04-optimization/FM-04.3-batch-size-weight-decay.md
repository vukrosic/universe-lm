# FM-04.3 — Batch size and weight decay scale with *data*, not model size

**Rule.** Optimal batch size and critical batch size scale as **power laws in the token budget
`D`, independent of model size `N`.** And optimal **weight decay scales linearly with batch
size** — so you cannot change batch size and keep the same `wd`. The unifying quantity is the
AdamW *timescale* `τ = B/(η·λ·D)`, which should stay roughly constant.

**Confidence.** **[C] Contested/Emerging.** A single rigorous recent scaling study, internally
validated, but not yet broadly independently replicated. Direction is well-motivated.

**Scope.** Bergsma et al. 2025 ("Power Lines"), dense transformer pre-training. The constant-`τ`
prescription assumes the AdamW + warmup-decay setup of FM-04.1/04.2.

## Evidence / details

- **B scales with D, not N.** Both `B_opt` (lowest loss) and `B_crit` (point past which more
  data-parallelism stops helping) follow power laws in `D`, *independent of N*. Practical
  consequence: size your batch from your token budget, not your parameter count.
- **wd scales with B.** Holding N, D fixed, optimal `λ ∝ B`. Reusing a `wd` across a batch-size
  change is a silent misconfiguration.
- **The invariant:** `τ = B/(η·λ·D)` ≈ const across settings; `τ_opt` itself follows a power
  law in the tokens/param ratio `D/N`.
- Caution: at very large batch (tens of thousands), AdamW can destabilize without careful LR
  tuning — `B_crit` is a real ceiling on useful data-parallelism.

## Reproduced by us?

**Held fixed, by RULE 0** — batch size and weight decay are explicitly out of the ledger's
structural scope. We hold them constant so structural Δs are clean. (Relevant only as a
warning: if we ever change batch size for an experiment, `wd` must move with it or the
comparison is confounded.)

## Falsifier

An in-scope sweep where `B_opt`/`B_crit` depend on `N` at fixed `D` (breaking the
data-only scaling), or where optimal `λ` is flat in `B`, would bound the rule.
