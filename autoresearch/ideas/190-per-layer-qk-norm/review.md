# Review log — 190 per-layer-qk-norm

## r1 — 2026-06-15 — verdict: revise
- **Mechanism premise is wrong.** 016's QK norm is `make_norm(self.d_k, "rmsnorm", _qk_use_ln)` — i.e. `nn.RMSNorm(d_head)` with `weight` of shape `(d_head,)` (see `RMSNorm` docstring at `models/layers.py:413` and `make_norm` at `models/layers.py:552-612`). The weight is **per-channel** (size `d_head=16`), broadcast across the H axis because heads are stacked along H and RMSNorm normalizes only the last dim. 016 has **no per-head γ axis**. Per block: 16 (Q) + 16 (K) = 32 params. Per run: 12 blocks × 32 = **384** params, not 48. The "one gain per head per block" framing in `idea.md:13/20-22/28-29` is a misread of `nn.RMSNorm(d_head)`. The "per-head γ axis is over-parameterized" bet doesn't even apply because 016 doesn't have a per-head γ to share.
- **The underlying lever is real, but it has to be reframed.** What 190 actually proposes is: replace 016's per-channel γ of size `d_head=16` with a **scalar γ** of size 1 per block (applied after RMSNorm, before the QK dot product). That is a real, novel lever on the QK-norm-attribution axis — it tests whether 016's WIN is driven by the per-channel resolution (16 scalars/block) or by a simpler block-level scalar magnitude (1 scalar/block). None of the closed 016/162/165/169 levers cover this. Reviser should rewrite `idea.md:20-29` to drop the "per-head vs per-block" framing and describe the lever as "scalar γ per block vs per-channel γ".
- **Param math must be corrected.** 016's QK-norm param count is 12 blocks × 2 (Q+K) × `d_head=16` = **384**. 190 with one scalar γ per block (separate Q and K) is 12 × 2 × 1 = **24**. With one scalar γ per block shared across Q and K it would be 12 × 1 = **12**. The "12 instead of 48" in `idea.md:20/28` and the `+0.013%` overhead in the description are both wrong; correct the math in the design sketch. (Negligible either way vs the 0.94M base — the point is the math, not the budget.)
- **Step-0 byte-identity claim survives, with corrected framing.** A scalar γ=1 applied after RMSNorm(Q) is still 1·x = x, so 190 with γ_b=1 is bit-identical to 016's per-channel weight all-ones init at step 0. But `idea.md:27` frames it as "γ=1 per head, which is the 016 init" — that part is wrong (it's γ=1 per channel of d_head). Rewrite the step-0 invariant in the correct terms.
- **Config flag name is misleading.** `qk_norm_share_across_heads: bool = False` (`idea.md:25`) suggests "share γ across heads," but the lever is a per-block scalar γ, not head-sharing. Rename to match the actual mechanism — e.g. `qk_norm_scalar_per_block: bool = False` — and add a separate flag if the design also wants Q/K to share the scalar (`qk_norm_scalar_qk_shared: bool = False`). Default both off, init both to 1.0 for step-0 identity.
- **Q-side vs K-side split must be specified.** 162 (Q-only NULL) and 165 (K-only NULL) closed the Q-side and K-side of the 016 attribution, leaving 016's WIN attributed to *joint QK symmetry*. 190 should preserve that symmetry, so the default should be **separate scalar γ for Q and K** per block (12 × 2 = 24 params), not a single shared scalar (12 params). The shared-scalar variant would be a different lever that would re-collapse to the 162/165 split. Reviser should call this out.
- **Attribution insight must be reframed.** The current "block-level vs head-level granularity" bet (`idea.md:34-35`) is built on the false premise. Replace it with: "is 016's WIN driven by the per-channel resolution of γ, or by the simpler block-level scalar magnitude? 190-WIN (matches 016's WIN magnitude with a single scalar γ per block per side) ⇒ scalar γ suffices, per-channel resolution is over-parameterized; 190-NULL (loses to 016 ctrl by >0.005) ⇒ per-channel resolution is binding, scalar γ throws away capacity." Either branch is informative.
- **Pass bar needs re-anchoring, not loosening.** With the corrected mechanism the pass bar should be:
  - **PASS**: trt ≤ 016-ctrl − 0.005 (matches 016's WIN magnitude under scalar γ) — confirms scalar γ is sufficient.
  - **NULL band |Δ| < 0.005** vs 016-ctrl — inconclusive (within noise; doesn't distinguish the levers).
  - **LOSS**: trt > 016-ctrl + 0.005 — the per-channel resolution is binding and 190 is a regression.
  Note: the taste-reviewer's `idea.md` finding (#1) suggested "Δ ≤ −0.01 vs the 016-equipped ctrl" — that's too loose. Tighten to the 016 plan's own bar (Δ ≤ −0.005 vs ctrl). One seed, one tier, so 0.005 is the realistic threshold.
- **A/B baseline is correct (compare to 016 ctrl).** The taste-reviewer's finding #2 stands in the corrected framing: trt should be `Tiny1M3MQKNormConfig` + 190 flag ON, ctrl should be `Tiny1M3MQKNormConfig` (016 WIN, val 6.3906 per `evidence.md`). The A/B must isolate the **granularity axis** (per-channel vs scalar γ), not the **existence axis** (RMSNorm vs no-RMSNorm) — comparing 190 to `Tiny1M3MConfig` would conflate 016's WIN with 190's null and tell us nothing about the lever. State this explicitly in the design sketch.
- **<200 LoC, no scope creep, single seed, tiny1m3m only** — all fine. The lever is a flag + a small branch in `MHA.__init__`/`forward` (~15 LoC). No conflicts on `models/layers.py` or `configs/llm_config.py` (`git diff` clean against the other AI's work).
- **Source citations are real** (016/162/165/169 all in-repo and verifiable in `closed.md`; Zhang & Sennrich RMSNorm arXiv:1910.07467 is the right 2019 paper). `transfer-risk: low` stays correct once the mechanism is reframed — scalar γ is a strictly coarser parameterization than per-channel γ, not a niche one.
- **Reviser action items, in order**: (1) rewrite `idea.md:13/20-29/34-35` to drop the "per-head vs per-block" framing and describe the lever as "scalar γ per block, per side"; (2) fix the param math to 384 (016) → 24 (190), not 48 → 12; (3) fix the step-0 invariant clause; (4) rename the config flag and add the Q/K-shared sub-flag; (5) specify separate Q/K scalar γ as the default to preserve 016's QK symmetry; (6) re-anchor the pass bar (PASS ≤ 016-ctrl − 0.005, NULL band |Δ| < 0.005, LOSS > 016-ctrl + 0.005); (7) call out the 016-ctrl baseline explicitly in the design sketch. Then resubmit as r2.
