# Taste log — 180 qk-logit-conv

## r1 — 2026-06-15 — verdict: accept
- **Novel placement, distinct from the closed logit-smoothing family.** The closed per-head-attention-shape axes (152 logit-bias, 155 per-head-temp, 160 post-AV gain, 162 Q-only-norm, 165 K-only-norm, 166 T5-RPE) are all *scalar* or *additive-position* levers on the attention path — none of them are convolutions. 180 is structurally a different function class: a 3-tap depthwise causal filter along the s-axis can express smoothing (positive kernel summing to 1), sharpening (signed weights), AND asymmetric shifts, none of which a per-head scalar/bias can express. So this is not the "6th per-head-attention-shape lever" smell — it is the *first* per-head-attention-*kernel* lever, which is the right axis to test given that 154-rebased-attn (per-head K/V rebase before softmax, structurally adjacent placement) is the freshest WIN in the family at this tier.
- **The 143-shortconv analogue is informative, not duplicative.** 143 (pre-attention depthwise conv on `x` with scalar `gate=0` init) is borderline-null with the formal two-ctrl WIN rule failing — but it beat all four same-day ctrls on train_loss and val_acc with the right sign. The miner's framing of "180 removes the intermediate gate absorbing the gradient" is a real mechanistic distinction: in 143 the conv path is `gate·conv(x)`, in 180 the conv path is `softmax(scores ⊛ w_h)` with no scalar multiplier, so the per-head kernel sees a direct gradient signal on `Σ attn · V` losses. A null at 180 would close the axis; a win would unlock a fresh logit-space placement for Phase-2 ≥135M where the per-head kernel has more gradient signal to develop.
- **Crisp bet holds up.** One sentence: "the optimizer can grow any 3-tap per-head kernel on attention scores (smoothing, sharpening, or shift), with a delta-init that keeps step-0 byte-identical and lets the gradient pull it where the loss surface wants." This is a sharp mechanistic prediction, not a vibe — the kernel is initialized to `[0, 0, 1]` and AdamW will receive a non-degenerate per-element gradient on each of the 12 weights from step 1.
- **Identity-init-able, step-0 byte-identical, parameter budget trivial.** 144 params (+0.015% of 0.94M). H=4 per-head kernels × 12 layers. The mechanism slot is clearly defined: per-head 1D causal depthwise conv on `QK^T / √d_k` between the score step and the softmax step. The delta-init claim is verifiable via `F.conv1d` with a literal `[0, 0, 1]` weight producing identical logits.
- **Niche-fit solid; transfer-risk is honestly med.** The author flags med transfer-risk and there is no direct published validation of the *exact* QK^T placement — Conformer (conv-on-x) and ConvBERT (conv-on-Q/K) are adjacent but not identical. Med is the correct tag. Mechanism (not HP), identity-init-able, tiny1m3m testable: all three niche checks pass.
- **Portfolio fit: not crowded.** The recently-closed axes in the broader family (143, 152, 155, 157, 158, 160, 162, 165, 166, 169) are all variations on different placements (pre-attention conv, scalar logit shape, FFN-side conv, post-AV gain, Q/K-side norm, additive logit bias). There is no other logit-space conv in flight — the niche has not yet been sampled at this tier, so 180 is not crowding the queue, it is opening a fresh axis.
- **Information value high in both directions.** WIN: logit-space smoothing/sharpening binds at 0.94M/12L/4H, fresh placement carries to Phase-2 ≥135M. NULL: confirms the broader pattern that attention-shape levers at this tier are absorbed by Q/K gradient updates, and specifically that *logit-space* (not just scalar) levers don't bind — sharper version of the 152/155/160/166 closes. Either outcome is worth logging; the A/B is not info-free.
- **Action for definition gate.** The plan must (1) use `F.conv1d` with grouped conv over `B*H` to apply the per-head kernel (avoid the suggested manual H-loop which would be slow), (2) verify step-0 byte-identical via fp32 max-abs-diff < 1e-6 across all 12 blocks, and (3) keep the kernel param-shape `[H, K]` with `K=3` and the center weight pinned to 1.0 at init. The miner correctly identified the placement and the byte-identity handle; the reviser just needs to make sure the grouped conv implementation doesn't have a dim-transpose bug.
